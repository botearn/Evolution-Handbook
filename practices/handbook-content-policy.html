Handbook Content Policy ​

状态：Draft

本规则定义 Evolution Handbook 的最小结构和内容质量门槛。

参与原则 ​

  • 团队所有成员都有权提出补充、修正、反例和废弃建议。
  • README、索引、链接和 Draft 不设隐性维护者门槛。
  • 内容是否合并按影响分级审查，而不是按贡献者职位判断。
  • 组织 Standard、治理和跨项目约束需要相关 Owner 明确批准。
  • 不完整但有价值的观察可以先进入 Issue 或 Draft PR。

具体参与流程见 CONTRIBUTING.md。

目录最小完整性 ​

每个长期存在的子目录应有 README.md，至少说明：

  1. 目录解决什么问题。
  2. 哪些内容应该或不应该进入。
  3. 当前已有文档及其状态。
  4. 新内容的准入条件。
  5. 已知缺口或下一步候选内容。

目录尚无正式内容时，应明确写“暂无已晋升内容”，不能用空目录暗示存在组织标准。

文档最小元数据 ​

原则、模式、Playbook、契约、决策和案例应尽量包含：

  • 状态：Draft、Pattern Candidate、Adopted、Standard、Deprecated 或 Superseded
  • 适用范围与不适用范围
  • Owner 或负责角色
  • 证据、来源或最后核对日期
  • 验证方式或复审触发条件

在负责人尚未确定时使用 Owner：待指定，不得猜测个人姓名。

结构化 metadata ​

契约、决策和项目 Playbook 可使用 YAML frontmatter，基础字段包括：

  • status：文档或决策当前状态
  • owner：负责角色、团队或 待指定
  • scope：organization、project、repository 或 domain
  • last_reviewed：最近核对日期

现有文档可以继续保留 **状态：...** 行，并在复审或实质修改时渐进迁移。metadata 只用于机器索引、复审和漂移检查；状态字段不代表自动晋升，Standard 仍需要相关 Owner 明确批准。

内容质量 ​

  • 原则解释“为什么”，不堆积具体命令。
  • Pattern 说明问题、适用条件、结构和代价。
  • Playbook 提供步骤、检查点、失败处理和完成条件。
  • Contract 明确 Owner、生产者、消费者、不变量和变更方式。
  • Decision 记录背景、选择、替代方案、影响和状态。
  • Case Study 同时记录结果、失败与限制。
  • Reference 说明采用了什么、没有采用什么，以及最后核对时间。

新增与复审 ​

新增文档时：

  1. 从 WORKSPACE.md 确认归属。
  2. 更新所在目录的 README.md。
  3. 新增项目或仓库时更新相关项目文档。
  4. 运行本地链接、格式和敏感信息检查。

编辑性修正可使用轻量 PR。约束性内容还应说明影响范围、Owner、迁移方式和验证证据。

以下情况触发复审：

  • 所引用的产品、工具或 API 发生重大版本变化
  • 实际项目行为与文档冲突
  • 同一规则出现两个长期真源
  • 实践连续两次未产生预期结果
  • Owner、项目状态或仓库边界变化

不接受的内容 ​

  • 无来源的“行业最佳实践”列表
  • 未脱敏的内部数据和本机信息
  • 从其他仓库整篇复制的实现文档
  • 没有退出条件的临时兼容方案
  • 只有宣传结论、没有证据和限制的案例
