# Contributing to Evolution Handbook

Evolution Handbook 由团队共同维护。任何成员都可以修正文档、补充实践、提交案例或挑战现有结论。

贡献的目标不是一次写出完美标准，而是让真实经验以可审查、可演进的方式进入组织知识库。

## 最快的参与方式

低风险修改可以走 fast path：小 PR、少说明、快速 Review。

适合 fast path：

- typo、格式、断链
- README 导航
- 索引同步
- 明确过期描述
- Draft 的限制、反例或小案例补充

不适合 fast path：

- Standard、治理、安全、权限
- 项目契约和 breaking change
- Registry 归属变化
- 涉及生产、结算、客户或私密数据的内容

其他修改可以直接建立分支并提交 Pull Request：

- 更新根目录或子目录 `README.md`
- 补充来源、术语和导航
- 更新已确认的项目或仓库状态
- 为现有文档增加反例、限制或验证证据

不确定归属或尚未形成完整结论时：

1. 提交 Knowledge Candidate Issue；或
2. 建立 Draft PR，在描述中写清已知事实和待讨论问题。

## 标准贡献流程

1. 阅读 `WORKSPACE.md`，判断内容属于组织、项目还是仓库。
2. 搜索是否已有真源，避免创建重复正文。
3. 从 `templates/` 选择最接近的模板；简单修订无需使用完整模板。
4. 新内容默认标记为 `Draft` 或 `Pattern Candidate`。
5. 更新所在目录的 `README.md` 索引。
6. 新增或调整项目、仓库时同步更新 `registry/`，并运行 `npm run validate:registry`。
7. 提交 PR，说明问题、变化、依据和验证。
8. 根据影响范围邀请相关 Reviewer。

## 分级审查

| 变更类型 | 示例 | 合并要求 |
|---|---|---|
| Level 0 编辑性 | 拼写、格式、链接、导航 | 任一团队成员审查 |
| Level 1 补充性 | Draft、Reference、模板、案例候选 | 至少一名相关领域成员审查 |
| Level 2 约束性 | Adopted Practice、项目契约、Registry 归属 | 相关 Owner 或受影响项目代表审查 |
| Level 3 组织性 | Standard、治理、安全、权限和跨项目默认规则 | 相关 Owner 明确批准，并列出影响与迁移方式 |

Reviewer 数量不是目的。关键是让承担影响的人参与审查，且高影响规则不能仅凭文案质量合并。

## PR 应回答

- 解决什么问题？
- 这是事实、建议、候选实践还是既有标准？
- 影响哪些项目、仓库或角色？
- 证据或来源是什么？
- 如何验证？有哪些限制或反例？
- 是否引入新的真源或需要迁移旧文档？

仓库提供了 Pull Request 模板，简单修改可以删除不适用项。

## 本地检查

提交 PR 前建议运行：

```bash
npm install
npm run check
```

当前检查覆盖 Markdown 基础格式和 Registry YAML 可解析性。CI 只是基础防线，不代表事实、来源、状态、Owner 和隐私边界已经完成核对；这些仍由提交者和 Reviewer 负责。

## README 也可以直接更新

README 是入口，不是只读封面。发现以下问题时请直接修改：

- 新成员不知道从哪里开始
- 目录结构已经变化
- 常用入口没有被链接
- 项目状态描述与事实不符
- 贡献方式不够清楚

README 只保存导航和稳定概述。较长规则应进入对应目录，再从 README 链接。

## 使用 Agent 贡献

可以使用 Codex、Claude Code 或其他 Agent 帮助整理内容，但提交者仍需：

- 核对事实、链接和项目边界
- 删除模型猜测和无法确认的结论
- 确认没有密钥、客户数据、本机路径或私密上下文
- 检查是否复制了受版权保护或由其他仓库维护的长篇内容
- 对最终修改和影响负责

Agent 生成不等于验证完成。PR 中应写明实际核对过什么。

## 内容放置

| 内容 | 位置 |
|---|---|
| 组织责任和治理 | `governance/` |
| 跨项目通用工程实践 | `practices/` |
| AI Native 原则、工具和工作流 | `ai-native/` |
| 项目边界和跨仓库契约 | `projects/<project>/` |
| 可复制结构 | `templates/` |
| 项目与仓库登记 | `registry/` |

更详细的路由以 `WORKSPACE.md` 和 `SOURCE_OF_TRUTH.md` 为准。

## 安全与隐私

本仓库是公共 Handbook。不得提交：

- Token、密钥、账号和生产凭据
- 客户、用户、员工的私密信息
- 未脱敏的会议记录、事故日志和经营数据
- 内部访问地址和本机绝对路径
- 未经授权公开的私有仓库内容

需要以私密证据支撑结论时，在 PR 中描述脱敏后的结论和证据类型，不复制原始数据。

## Review 指南

Reviewer 优先检查：

1. 内容是否放在正确层级。
2. 事实、建议和待验证假设是否区分。
3. 是否出现重复真源或过度复制。
4. 状态是否与证据匹配。
5. 受影响的 Owner 和项目是否知情。
6. 链接、Registry 和目录索引是否同步。
7. 是否包含不应公开的信息。

Review 可以挑战结论，但应给出原因、反例或更合适的归属建议。

## 合并后的维护

文档不是合并后永久有效。遇到实现冲突、工具升级、项目边界变化或新反例时，任何成员都可以发起后续修订、降级或废弃。
