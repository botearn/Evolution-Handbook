# Context as Infrastructure

**状态：Draft**

AI Native 组织需要把上下文当作基础设施和竞争力，而不是临时塞进 Prompt 的背景材料。

同样的模型，在模糊指令和高质量 Context 下会产生完全不同的结果。组织的长期优势不只来自采用了哪种 AI 工具，也来自它能否持续为人和 Agent 提供准确、相关、可追溯的 Context。

## 原则

- 关键上下文必须可发现、可引用、可更新。
- 同一个事实只维护一个真源。
- 给人的文档和给 Agent 的指令应共享事实层，而不是各写一套。
- 上下文应靠近其拥有者：组织原则在 Handbook，项目契约在项目层，代码规则在仓库内。
- 临时对话不能成为长期决策的唯一载体。

## 好的上下文具备

- 明确所有者
- 明确适用范围
- 可判断是否过期
- 能链接到证据或可执行资产
- 能被人和机器稳定定位

## Context Architecture 三层

### Context Org Chart

定义一个任务或项目需要哪些 Context，以及谁负责产生和维护。

它回答：

- 关键 Context 散落在哪里？
- 哪些角色拥有这些信息？
- 谁负责更新，什么时候过期？
- 哪些内容默认暴露，哪些内容按需披露？

这里的 Org Chart 不是传统汇报关系图，而是 Context 的生产、所有权和流动关系图。

### Context Architecture

设计 Context 从原始信息到可复用知识的流转方式：

```text
原始证据
  -> 结构化 Observation
  -> 可检索项目知识
  -> Agent 可消费的指令、Skill 和 Contract
  -> 真实任务验证
  -> 新证据
```

它需要解决真源、版本、权限、检索、渐进式披露和知识晋升。

### Context Toolchain

选择支撑 Context 流转的工具，例如：

- 项目和组织文档
- 代码仓库与 `AGENTS.md`
- 飞书知识库和协作 Thread
- OpenClaw、Codex 和 Claude Code
- MCP、CLI、搜索和结构化 Registry
- Observation、Reflection 和评估工具

工具链服务于 Context Architecture。不能因为某个工具方便，就让它成为未经设计的新真源。

## 反模式

- 同一规则复制到多个 `CLAUDE.md`、`AGENTS.md` 和 Wiki
- 用超长 Prompt 代替结构化文档
- 把聊天记录当作项目唯一历史
- 文档描述理想架构，但不链接真实实现
- 把全部历史无差别推入每次 Session
- 只建立规则库，却没有从真实工作持续产生新规则的机制

## 当前实践

Evolution Handbook 使用组织、项目、仓库三级真源，并通过引用而非复制连接它们。

这仍是待验证的组织实践。完成至少一个项目的持续采用和复盘后，再评估是否升级为 `Adopted`。

## 竞争力判断

个人价值不会简单变成“谁提供的 Context 最多”。真正有价值的是：

- 能识别任务所需的关键 Context
- 能判断来源可信度和时效
- 能把隐性知识转成可复用结构
- 能控制隐私、权限和无关信息
- 能让 Context 在执行后被验证和更新

Context 越多不一定越好。相关、准确、及时和可操作比堆积信息更重要。

## 闭环要求

上下文基础设施至少包含：

```text
原始证据
  → 观察
  → 反思与合并
  → 原则、契约、Skill 或 Playbook
  → 在真实任务中验证
  → 新证据
```

只有存储，没有晋升和淘汰，不构成可持续记忆系统。
