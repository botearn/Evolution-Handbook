# 可回看的 Agent 工作台

**状态：Pattern Candidate**

**Owner：待指定**

## 问题

Agent 工作很快，但如果只留下最后结果，后来很难判断：

- 当时为什么这么改？
- 哪一步是人批准的？
- 哪些检查真的跑过？
- 失败或中断发生在哪里？

## 模式

把 Agent 工作台当成“可以回看的协作现场”，而不是一次性聊天窗口。

推荐做法：

- 复杂任务先写计划，再执行。
- 关键动作留下检查点：开始、用工具前后、权限请求、子任务开始/结束、收尾。
- 人类审批和普通自动检查分开处理；审批可以等久一点，普通检查要快失败。
- 默认少外联，敏感信息只留在本地或密钥系统。
- 最终回答写清楚做了什么、怎么验证、还剩什么风险。

## 何时采用

适合：

- 多文件修改
- 跨仓协作
- 高权限工具
- 需要人类审批的任务
- 之后可能要复盘的工作

## 不适用

不适合把每个小动作都包装得很重。错别字、轻量链接修正、小范围 README 调整，可以直接改、直接验。

## 代价与风险

- 配置太重会打断节奏。
- 记录太细会让人不想看。
- 本地 hook、日志和计划不能泄露密钥、内部地址或私密上下文。

## 验证

一次任务结束后，应该能快速回答：

- 当时的计划是什么？
- 哪些动作需要人批准？
- 跑过哪些检查？
- 如果结果出问题，应从哪里回看？

## 采用证据

- 项目：Evolution Handbook
- 结果：本仓开始使用计划文件、`npm run check`、Registry 校验和 metadata 校验来保留可回看的工作轨迹。
- 限制：当前仍是 Pattern Candidate，尚未证明适用于所有项目。
