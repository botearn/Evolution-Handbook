# ARTI AI Native Adoption

**状态：Draft**

本文件记录 ARTI 如何采用组织级 AI Native 实践，以及项目特有的例外。

## 当前成熟度

| 能力 | 状态 | 说明 |
|---|---|---|
| 代码附近的 Agent 指令 | 部分采用 | 四个仓库的规则文件不一致 |
| RFC 驱动的重要决策 | 已采用 | 各仓库已有 RFC，但跨仓状态维护不统一 |
| 确定性验证 | 部分采用 | 有测试和契约校验，跨仓漂移检查仍不足 |
| 高风险动作人工批准 | 已采用 | Push、部署和生产操作默认需明确授权 |
| Skill 真源与 Bridge | 未建立 | 尚未形成项目级 `skills/` 真源 |
| 组织记忆 | 建设中 | Handbook 开始承担项目级索引和稳定契约 |

## 已采用实践

- 组织、项目、仓库三级上下文
- 仓库本地规则贴近代码维护
- 非琐碎架构改动使用 RFC
- 高影响远程操作需要人类明确批准
- 测试和运行结果作为完成证据
- 共享数据库 migration 采用 append-only
- 失败、跳过和未验证项需要显式暴露

## 近期建设顺序

1. 为 `ARTI_backend` 和 `ARTI-CLI` 补齐平台无关的 `AGENTS.md`。
2. 为 `report_tasks`、通知、Credits 和共享枚举增加跨仓漂移检查。
3. 建立项目级 Skill 候选清单，再决定是否创建 `skills/` 真源。
4. 把跨仓 RFC 的关系做成可检查索引。
5. 登记 Agent 可执行的远程操作及其批准级别。

## 项目例外

ARTI 涉及金融数据和用户资产相关逻辑。错误内容可能比缺失内容更有害，因此：

- 数据覆盖不足时不得静默返回相似内容
- 资金、积分和结算相关变更需要服务端权威校验
- 不可逆的生产操作必须保留人工批准

具体实现规则以对应仓库文档为准。

## 可沉淀的内部案例

- 共享 `settle_market` RPC 被单侧旧版本覆盖，导致通知逻辑回归
- 客户端扣费形成“假门”，迁移为服务端权威
- 跨域双 Session 导致认证刷新冲突，改为同源与单一存储
- Worker 旧路径长期存在于文档，排障时必须先确认真实生产入口
