# Cross-Repository Change

## 适用场景

- 一个行为需要修改两个以上仓库
- 修改共享枚举、任务、结果或 API 契约
- 改动会影响主站、Backend、Poly 或 CLI 的组合

## 步骤

### 1. 定义行为

先写用户可判断的结果：

- 什么行为会改变
- 什么必须保持不变
- 如何证明完成

不要以“修改四个文件”作为成功标准。

### 2. 画影响图

列出：

- Producer
- Contract Owner
- Consumer
- 数据库对象
- 部署目标
- 回滚入口

### 3. 选择主篇

按主要决策和改动量选择 RFC 主篇，其余仓库建立姊妹篇。

主篇负责：

- 动机
- 共同设计
- 权衡
- 全局发布顺序

姊妹篇负责：

- 本仓库实施
- 本地测试
- 本地回滚

### 4. 确认真源

对每个共享对象写清：

- 当前 Owner
- 当前生产定义
- 谁只消费
- 是否有历史副本

发现冲突时不要平均两份实现。以生产真身、更新版本和测试证据选定一个。

### 5. 设计兼容期

需要时使用：

- 先加后删
- 双读单写
- 双写校验
- Read-side normalize
- Feature flag

每个临时兼容层必须有退出条件。

### 6. 分仓实施

推荐顺序：

```text
契约与数据层
  → 服务端 Producer
  → Consumer
  → 清理旧路径
```

若新 Consumer 可以兼容旧 Producer，可以先发 Consumer。

### 7. 验证

至少覆盖：

- 每个仓库的本地测试
- 契约测试
- 一个端到端真实路径
- 失败与重试路径
- 历史数据兼容

### 8. 更新项目记忆

完成后：

- 更新双方 RFC 状态和 PR
- 更新本 Handbook 契约或决策索引
- 删除已经退出的临时兼容说明

## 完成标准

- [ ] 主篇与姊妹篇互引
- [ ] Owner 明确
- [ ] 发布和回滚顺序明确
- [ ] 各仓测试通过
- [ ] E2E 路径验证
- [ ] 临时兼容层有退出条件
- [ ] Handbook 已更新
