## 1. 产品摘要

### 1.1 一句话定义

ARTI 是一个 AI 驱动的投研与决策验证平台，帮助用户从“发现市场信号”进入“获得分析、形成判断、采取模拟行动、持续追踪和复盘”的完整闭环。

### 1.2 用户问题

目标用户面临的不是“没有金融信息”，而是：

- 信息过多，无法快速判断哪些与自己相关。
- 单一观点缺少交叉验证，难以建立信任。
- 分析结束后没有行动承接，容易看完即走。
- 判断结果没有持续追踪，无法知道自己和 AI 是否真的有效。
- 报告、关注、预测和账户信息分散，难以形成长期记忆。

### 1.3 产品承诺

ARTI 应让用户：

1. 快速看懂一个标的或事件现在发生了什么。
2. 理解 AI 的判断、证据、分歧和风险。
3. 选择继续深入、加入关注或用模拟 Credits 验证判断。
4. 在报告完成、价格变化或结果结算时被召回。
5. 在记忆和历史中看到长期判断质量，而不是只消费一次内容。

### 1.4 非承诺

ARTI 不承诺：

- 保证投资收益或预测正确。
- 替代持牌投资顾问。
- 提供真实资金交易或提现。
- 把 AI 输出包装成无风险结论。
- 在数据不足时生成看似完整的相似答案。

所有面向用户的运营文案必须明确：内容用于研究与决策辅助，Credits 为产品内模拟积分。

## 2. 当前产品阶段

### 2.1 阶段判断

ARTI 已具备较完整的功能面，但尚未达到“运营可以独立放量”的状态。

| 能力 | 当前判断 |
|---|---|
| 用户注册与身份 | 已有 |
| 快速诊断 | 已有 |
| 全景与深度报告 | 已有，但可靠性和耗时需持续治理 |
| 关注与投资记忆 | 已有 |
| 预测市场与模拟下注 | 已有，仍存在服务端权威化缺口 |
| Credits 与订阅 | 已有，定价和权限属于高风险配置 |
| 站内通知 | 已有 |
| 邮件摘要 | 正在迁移和验收 |
| 运营后台 | 功能较多，但入口复杂、权限粒度不足 |
| 数据与质量评估 | 已有多种看板，定义尚未完全统一 |
| 稳定发布门禁 | 未完全恢复 |

### 2.2 当前运营原则

在可靠性基线完成前：

- 可以继续种子用户访谈、内容运营、低风险体验优化和数据基线采集。
- 不建议大规模投放或承诺稳定 SLA。
- 不把 Credits、结算、邮件摘要或跨仓切流混入普通活动发布。
- 每次活动必须有人工观察、停止条件和用户补偿方案。

## 3. 目标用户

### 3.1 核心用户

#### A. 需要快速形成判断的个人投资者

特征：

- 关注 A 股、港股或美股。
- 有一定市场经验，但没有完整研究团队。
- 希望先看结论，再选择是否深入。

核心需求：

- 快速诊断。
- 多维证据和风险提示。
- 目标价、入场区间、止损等可执行表达。
- 关注与后续验证。

#### B. 对事件和概率感兴趣的决策型用户

特征：

- 关注宏观、科技、体育或公共事件。
- 能理解概率，但不一定熟悉订单簿或 Web3。
- 愿意用模拟 Credits 验证判断。

核心需求：

- 看懂市场概率代表什么。
- 比较市场定价与 AI 判断。
- 通过模拟下注形成反馈。
- 在结算后看到结果和历史表现。

### 3.2 次级用户

#### C. 专业研究和工具用户

通过 CLI 或 MCP 使用 ARTI，重视结构化输出、可组合性和稳定接口。

#### D. 内部运营与质量团队

需要管理用户、活动、内容、Credits、报告任务、反馈和模型质量。

### 3.3 暂不优先

- 以高频交易和真实下单为主的专业交易员。
- 需要机构级审计、组合管理和合规报表的企业客户。
- 只消费泛财经资讯、不产生判断或行动的流量用户。

## 4. 核心产品闭环

```mermaid
flowchart LR
    A["发现<br/>市场、资讯、关注"] --> B["理解<br/>快速诊断"]
    B --> C["深入<br/>全景或深度报告"]
    B --> D["行动<br/>关注、模拟下注"]
    C --> D
    D --> E["追踪<br/>记忆、持仓、报告"]
    E --> F["反馈<br/>通知、结算、准确率"]
    F --> A
```

### 4.1 闭环定义

一次有效决策闭环至少包含：

1. 用户获得一份有效 AI 分析结果。
2. 用户执行一个可追踪行动。
3. 用户在结果变化、报告完成或结算后回来查看。

可追踪行动包括：

- 加入关注。
- 生成更深一档报告。
- 使用模拟 Credits 下注。
- 保存或再次查看报告。

### 4.2 新用户黄金路径

当前产品选择的最小激活路径：

```text
注册
  → 完成一次快速诊断
  → 进入预测或判断场景
  → 完成一次模拟下注
  → 在通知、持仓或历史中查看结果
```

快速诊断和模拟下注是当前 onboarding 的两个数据库真值任务。

### 4.3 股票研究路径

```text
输入股票或问题
  → 快速诊断
  → 查看结论、证据与风险
  → 加入关注或生成全景报告
  → 需要时生成深度研究
  → 报告异步完成通知
  → 在记忆中追踪目标与实际表现
```

### 4.4 预测市场路径

```text
浏览事件
  → 理解当前概率与决议规则
  → 查看 AI 判断与市场偏差
  → 使用模拟 Credits 选择 Yes / No
  → 查看持仓与概率变化
  → 市场结算
  → 收到结果通知并复盘
```

## 7. 运营工作台需求

### 7.1 运营首页

运营首页应在一屏内回答：

- 今天有多少新用户、激活用户和闭环用户？
- 报告和预测链路是否健康？
- 是否有 Credits、结算或通知异常？
- 哪个漏斗环节下降最多？
- 今天需要人工处理什么？

建议首页模块：

| 模块 | 指标或内容 | 更新频率 |
|---|---|---|
| 用户概览 | 新增、活跃、激活、闭环 | 日 |
| 核心漏斗 | 注册 → 快速诊断 → 深入 / 下注 → 回访 | 日 |
| 报告健康 | 请求、成功、失败、P50/P95、卡死 | 实时 / 日 |
| Credits 健康 | 发放、消耗、退款、冻结异常 | 日 |
| 通知健康 | 创建、发送、失败、点击 | 日 |
| AI 质量 | 用户反馈、评估分、明显错误 | 日 / 周 |
| 待处理 | 失败任务、异常用户、内容审核、告警 | 实时 |

### 7.2 用户支持

运营需要：

- 按邮箱、用户 ID 或邀请来源搜索用户。
- 查看套餐、Credits、报告、持仓和关键事件。
- 查看失败原因和可重试入口。
- 在授权范围内补发 Credits 或延长权益。
- 记录每次人工操作的原因、操作者和关联工单。

代理登录、重置数据、调账和改套餐属于高风险操作：

- 不能用于普通数据浏览。
- 必须展示目标用户和影响范围。
- 必须有二次确认和审计记录。
- 不能代理或重置其他管理员。

### 7.3 活动与邀请

运营应能配置：

- 活动名称、渠道、有效期和目标人群。
- 邀请码数量、使用次数和授予权益。
- 是否发放额外 Credits。
- 活动页文案和注册 CTA。
- 活动停止条件。

活动复盘至少包含：

- 访问、注册、激活、首次分析、首次行动。
- 7 日回访。
- Credits 发放、消耗和异常。
- 用户反馈和支持成本。

### 7.4 内容运营

每条公开内容需要：

- 明确目标用户。
- 对应一个核心用户问题。
- 至少一个产品内 CTA。
- 标注数据时间和来源。
- 区分事实、AI 判断和运营观点。
- 设置复核日期或失效条件。

推荐 CTA：

- 对标的做快速诊断。
- 加入关注。
- 查看相关预测事件。
- 阅读完整报告。

### 7.5 质量与反馈

反馈处理状态：

```text
New
  → Triaged
      → Product / Data / AI / Support
          → Fixed / Explained / Won't Fix
              → Closed
```

运营负责问题分类和用户沟通，不直接修改 Prompt、模型、定价或生产数据。

每周应输出：

- Top 用户问题。
- Top AI 错误类型。
- Top 数据缺失类型。
- Top 体验摩擦。
- 已修复与待决策事项。

## 8. 运营流程

### 8.1 每日检查

1. 查看报告成功率、队列深度和卡死任务。
2. 查看 Credits 发放、消耗、退款和冻结异常。
3. 查看结算与邮件摘要 Cron 结果。
4. 查看新用户激活漏斗。
5. 处理用户反馈和失败任务。
6. 审核当天准备发布的内容。
7. 在飞书 Thread 汇总异常、Owner 和关闭条件。

### 8.2 每周复盘

1. 查看核心闭环和留存趋势。
2. 按渠道、用户类型和产品入口分组。
3. 抽样检查快速诊断、全景和深度报告质量。
4. 分析 Credits 发放与消耗。
5. 复盘通知点击和召回效果。
6. 选择下周最多一个主实验和一个可靠性主题。
7. 更新 PRD、状态页和任务 Owner。

### 8.3 活动上线

```text
定义目标与人群
  → 确认权益和预算
  → 配置邀请 / 内容 / CTA
  → Dry Run
  → 人工审批
  → 小流量发布
  → 观察漏斗和异常
  → 扩量或停止
  → 复盘
```

### 8.4 事故与异常

需要立即停止相关活动：

- 余额、冻结或结算不一致。
- 重复扣费、重复发奖或重复邮件。
- 报告大面积失败或明显错误标的。
- 用户可看到其他用户数据。
- 高风险配置被未授权修改。

运营不直接做数据库修复。应保存现象、用户范围、时间、入口和截图，升级给系统 Owner。

## 9. 指标体系

### 9.1 北极星指标

**周可验证决策用户数（Weekly Verified Decision Users，WVDU）**

定义：一周内完成“有效 AI 结果 → 可追踪行动 → 后续回访或结果查看”的去重用户数。

为什么选择：

- 覆盖完整 ARTI，而不是只覆盖预测市场。
- 强调用户获得价值并形成长期反馈。
- 避免只优化注册、消息量或模型调用次数。

### 9.2 领先指标

| 指标 | 定义 |
|---|---|
| 新用户激活率 | 注册后 48h 内完成快速诊断和首次模拟下注 |
| 有效结果率 | 获得可展示结果的分析请求 / 全部分析请求 |
| 分析到行动率 | 获得结果后 24h 内关注、深入或下注的用户 / 获得结果用户 |
| 报告升档率 | 快速诊断后 7 日内生成全景或深度报告 |
| 通知召回率 | 点击通知并进入目标页面的用户 / 通知展示用户 |
| 7 日回访率 | 激活用户在第 7 日窗口内再次使用核心功能 |

### 9.3 护栏指标

| 维度 | 指标 |
|---|---|
| 正确性 | 非法标的拒绝率、错误市场归类数 |
| 稳定性 | 报告成功率、卡死任务数、API 可用率 |
| 资产 | 重复扣费数、异常退款数、冻结余额漂移 |
| 通知 | 重复通知率、邮件失败率、退订率 |
| 质量 | 明显幻觉率、负反馈率、数据缺失率 |
| 成本 | 单份报告 token、模型成本、缓存命中率 |

### 9.4 指标口径要求

- 每个指标必须有事件、分母、时间窗、去重键和 Owner。
- 不把页面浏览量当作核心价值。
- 不把模型调用次数当作用户价值。
- 旧 ARTI-Poly 指标只作为历史参考，不直接作为完整产品目标。
- 商业目标在完成连续 7 天基线采集后由产品与运营共同设定。

### 9.5 基线阶段必采事件

| 事件 | 说明 |
|---|---|
| `signup_completed` | 完成注册 |
| `quick_scan_completed` | 快速诊断产生有效结果 |
| `report_requested` | 发起全景或深度报告 |
| `report_completed` | 报告完成 |
| `report_failed` | 报告失败 |
| `watchlist_added` | 加入关注 |
| `prediction_bet_completed` | 真实写入模拟持仓 |
| `notification_clicked` | 点击通知 |
| `result_revisited` | 再次查看报告、记忆或结算结果 |

现有埋点分散在主站和预测市场实现中。V1 需要统一事件命名和用户去重口径。

## 10. 版本路线

路线按退出条件推进，不以日期强行切换。

### R0：可靠性基线

**目标**：产品可以被团队可信地验证和小规模运营。

范围：

- 修复非法 symbol fail-open。
- 恢复主站 lint、test、build 门禁。
- 完成预测市场 migration、结算和邮件摘要验收。
- 业务部分失败能触发告警。
- 修复阻断性依赖安全问题。

退出条件：

- P0 执行项关闭。
- Credits、结算和通知完成故障注入。
- 核心路径可重复 E2E。
- 运营有明确的停止活动按钮或流程。

### R1：运营可用闭环

**目标**：运营能独立完成日常观察、用户支持和小活动。

范围：

- 统一注册、快速诊断、报告、下注和回访漏斗。
- 运营首页聚合核心指标和待处理事项。
- 用户支持操作增加审计。
- 通知邮件稳定运行。
- 活动和邀请来源可追踪。
- 反馈有状态、分类和 Owner。

退出条件：

- 连续 7 天数据口径稳定。
- 日常检查不依赖工程师手工查库。
- 每项高风险操作有审批和审计。
- 运营完成一次小活动并输出复盘。

### R1.1：留存与实验

**目标**：从“能用”进入“用户愿意回来”。

范围：

- 关注和通知召回优化。
- 报告升档和复访引导。
- 内容到产品 CTA 归因。
- Cohort 和渠道分析。
- 每周单变量实验机制。

退出条件：

- WVDU 和 7 日回访可稳定计算。
- 至少两个实验完成全周期复盘。
- 通知频控和退订机制稳定。

### R2：规模化与商业化准备

**目标**：在安全边界内扩大用户和收入实验。

前置条件：

- Prediction Gateway 和 Prompt 真源收口。
- 预测成交参数服务端权威化。
- 后台具备细粒度 RBAC。
- 容量、成本和 SLO 有真实基线。
- 订阅、支付和退款经过合规与财务评审。

可能范围：

- 套餐和权益实验。
- 支付与订阅管理。
- Web Push 或更多召回渠道。
- 团队版、API 或专业工具能力。

## 11. 角色与权限

目标角色模型：

| 角色 | 可读 | 可写 | 禁止 |
|---|---|---|---|
| Ops Viewer | 看板、用户摘要、内容和反馈 | 无高风险写入 | 调账、改价、改模型 |
| Content Operator | 内容与活动数据 | 草稿、上下架、活动配置 | 调账、生产配置 |
| Support Operator | 用户、任务和流水 | 有审计的补偿与重试 | 改模型、批量数据修复 |
| Product Operator | 漏斗、Feature Flag、实验 | 低风险灰度 | 生产 migration、密钥 |
| Finance / Asset Owner | Credits、套餐、结算 | 经审批调账和定价 | 无审计操作 |
| System Admin | 全部 | 高风险系统操作 | 绕过审批和审计 |

当前后台主要依赖管理员身份，尚未完整实现上述 RBAC。R1 前必须至少将只读、内容、支持和资产操作分离。

## 12. 运营文案规范

必须：

- 区分“市场概率”“AI 判断”和“用户模拟持仓”。
- 明确 Credits 不可提现、不等于真实货币。
- 报告失败时说明退款或重试状态。
- 数据缺失时说明缺失，不用模型补齐具体数字。
- 活动权益写明有效期和适用范围。

禁止：

- “稳赚”“必涨”“100% 准确”等承诺。
- 将历史表现描述为未来保证。
- 将本地模拟记录描述为真实持仓。
- 用紧迫性文案诱导用户忽略风险。
- 未经审核自动公开高风险金融结论。

## 13. 开放决策

| 决策 | 为什么重要 | 建议 Owner | 截止阶段 |
|---|---|---|---|
| 核心目标用户先聚焦股票还是事件预测 | 决定首页、内容和增长入口 | Product | R0 结束前 |
| 模拟下注是否长期作为 onboarding 必做项 | 影响不关心预测市场的股票用户 | Product + Ops | R1 设计前 |
| `ARTi-poly` 独立站是否保留 | 影响运营入口、数据和维护成本 | Architecture + Product | R1 前 |
| 套餐、支付和真实商业目标 | 涉及合规、退款和权益 | Product + Finance | R2 前 |
| 准确率和“命中”统一口径 | 影响用户信任与运营宣传 | Product + Data | R1 前 |
| 邮件摘要的发送时区与频率 | 影响召回和退订 | Ops | R1 前 |

## 14. 迭代机制

### 14.1 需求状态

| 状态 | 含义 |
|---|---|
| Candidate | 用户问题或运营观察 |
| Draft | 已写需求，尚未批准 |
| Ready | Owner、指标、验收和依赖完整 |
| In Progress | 正在实施 |
| Experiment | 小流量验证 |
| Shipped | 已发布并完成回读 |
| Learned | 已完成数据和用户复盘 |
| Deprecated | 不再继续 |

### 14.2 每项迭代必须回答

- 解决哪个用户问题？
- 改变哪个核心流程？
- 影响哪个指标？
- 有什么护栏和停止条件？
- 需要哪些运营配置或用户沟通？
- 如何回滚？
- 发布后何时复盘？

### 14.3 实验卡模板

```text
实验名称：
用户问题：
假设：
目标人群：
唯一变量：
主指标：
护栏指标：
样本与周期：
停止条件：
发布方式：
结论：
后续动作：
```

同一周最多运行一个影响核心闭环的主实验，避免多个变量互相污染。

### 14.4 PRD 更新规则

- 小文案和范围澄清增加补丁版本，例如 v0.1.1。
- 新模块或版本路线变化增加次版本，例如 v0.2。
- 北极星、目标用户或商业模式变化增加主版本，例如 v1.0。
- 不删除历史决策，使用版本记录说明变化原因。
- 实现事实变化时，同时更新状态页和相关契约。
